Spring Cloud入门

微服务

1 概述

Spring Cloud 是微服务系统架构的一站式解决方案，在平时我们构建微服务的过程中需要做如 服务发现注册 、配置中心 、消息总线 、负载均衡 、断路器 、数据监控 等操作，而 Spring Cloud 为我们提供了一套简易的编程模型，使我们能在 Spring Boot 的基础上轻松地实现微服务项目的构建。

主要组件：

  • Eureka：服务治理组件，包含服务注册中心，服务注册与发现机制的实现。（服务治理，服务注册/发现）
  • Zuul：网关组件，提供智能路由，访问过滤功能
  • Ribbon：客户端负载均衡的服务调用组件（客户端负载）
  • Feign：服务调用，给予Ribbon和Hystrix的声明式服务调用组件 （声明式服务调用）
  • Hystrix：容错管理组件，实现断路器模式，帮助服务依赖中出现的延迟和为故障提供强大的容错能力。(熔断、断路器，容错)

2 组件

2.1 Eureka

Eureka 主要分为三个角色：注册中心（Eureka Server)，服务提供者（Eureka Client Provider），服务消费者（Eureka Client Consumer），类似的服务发现还有Zookeeper，Consul。

Eureka 客户端从服务器获取注册表信息，并将其缓存在本地。客户端会使用该信息查找其他服务，从而进行远程调用。该注册列表信息定期（每30秒钟）更新一次。每次返回注册列表信息可能与 Eureka 客户端的缓存信息不同, Eureka 客户端自动处理。如果由于某种原因导致注册列表信息不能及时匹配，Eureka 客户端则会重新获取整个注册表信息。 Eureka 服务器缓存注册列表信息，整个注册表以及每个应用程序的信息进行了压缩，压缩内容和没有压缩的内容完全相同。Eureka 客户端和 Eureka 服务器可以使用JSON / XML格式进行通讯。在默认的情况下 Eureka 客户端使用压缩 JSON 格式来获取注册列表的信息。正常情况下，如果 Eureka Server 在90秒没有收到 Eureka 客户的续约，它会将实例从其注册表中删除。

2.1.1 操作

  1. 创建Eureka Server，可以通过idea内置快速搭建spring boot项目功能创建Eureka Server。导入SpringClooud依赖，在配置文件中配置 server.port，spring.application.name，eureka.client.service-url.defaultZone(如http://127.0.0.1:${server.port}/eureka)。在引导类上加上@EnableEurekaServer注解。

  2. 不管是提供者还是消费者都是属于客户端的角色，注意点都差不多，导入依赖后，配置文件中都要配置：server.port，spring.application.name，eureka.client.service-url.defaultZone，引导类上加入@EnableDiscoveryClient注解。

​ 3.做高可用的Eureka Server的话可以使用互相注册的方式实现双Eureka Server，是两个Eureka Server的eureka.client.service-url.defaultZone为对方的 server.port，而在客户端的eureka.client.service-url.defaultZone中写上所有Eureka Server的地址。

2.2 Ribbon

Eureka集成了Ribbon可以通过直接修改代码使用Ribbon。通过@LoadBalanced注解来使用。（在消费方）

负载均衡策略可以通过配置文件中调整：

providerName:
  ribbon:
    NFLoadBalancerRuleClassName: #选一种
    com.netflix.loadbalancer.RandomRule #随机
    com.netflix.loadbalancer.RoundRobinRule #轮询

2.3 Hystrix

2.3.1 操作

1.引入hystrix依赖，在消费方引导类加@SpringCloudApplication注解（@SpringbootApplication，@EnableDiscoveryClient，@EnableCircuitBreaker三个注解的组合注解）

2.在消费方Controller的方法上加 @HystrixCommand(fallbackMethod = "ControllerMethod")注解，而在Controller加上同名同参同返回值的方法。

3.当然也可以设置默认的fallback方法，不过要在Controller类上加@DefaultProperties(defaultFallback = "defaultFallBack")，在类上指明统一的失败降级方法

4.hystrix.command.default.execution.isolation.thread.timeoutInMilliseconds来设置Hystrix超时时间。

2.3.2 熔断状态

熔断状态机3个状态：

  • Closed：关闭状态，所有请求都正常访问。
  • Open：打开状态，所有请求都会被降级。Hystix会对请求情况计数，当一定时间内失败请求百分比达到阈值，则触发熔断，断路器会完全打开。默认失败比例的阈值是50%，请求次数最少不低于20次。
  • Half Open：半开状态，open状态不是永久的，打开后会进入休眠时间（默认是5S）。随后断路器会自动进入半开状态。此时会释放部分请求通过，若这些请求都是健康的，则会完全关闭断路器，否则继续保持打开，再次进行休眠计时。

2.4 Feign

Feign可以把Rest的请求进行隐藏，伪装成类似SpringMVC的~一样。不用再自己拼接url，拼接参数等等操作，一切都交给Feign去做。（feign已经自动集成了Ribbon负载均衡的RestTemplate）

2.4.1 操作

1.消费方导入openfeign依赖

2.启动类上加入@EnableFeignClients 注解

3.创建一个接口

@FeignClient(value = "providerNmae") // 标注该类是一个feign接口
public interface UserClient {

    @GetMapping("user/{id}")
    User queryById(@PathVariable("id") Long id);
}

4.在Controller中调用

@Controller
@RequestMapping("consumer/user")
public class UserController {

    @Autowired
    private UserClient userClient;

    @GetMapping
    @ResponseBody
    public User queryUserById(@RequestParam("id") Long id){
        User user = this.userClient.queryUserById(id);
        return user;
    }
}

2.4.2 对Hystrix的支持

1.消费中配置：

feign:
  hystrix:
    enabled: true # 开启Feign的熔断功能

2.创建类实现有@FeignClient的接口

@Component
public class UserClientFallback implements UserClient {

    @Override
    public User queryById(Long id) {
        User user = new User();
        user.setUserName("服务器繁忙，请稍后再试！");
        return user;
    }
}

3.修改有@FeignClient的接口

@FeignClient(value = "service-provider", fallback = UserClientFallback.class) // 标注该类是一个feign接口
public interface UserClient {

    @GetMapping("user/{id}")
    User queryUserById(@PathVariable("id") Long id);
}

2.5 Zuul

不管是来自于客户端（PC或移动端）的请求，还是服务内部调用。一切对服务的请求都会经过Zuul这个网关，然后再由网关来实现 鉴权、动态路由等等操作。Zuul就是我们服务的统一入口。

2.5.1 操作

1.配置：

server:
  port: 10010 #服务端口
spring:
  application:
    name: api-gateway #指定服务名
eureka: #要加入Eureka依赖
  client:
    registry-fetch-interval-seconds: 5 # 获取服务列表的周期：5s
    service-url:
      defaultZone: http://127.0.0.1:10086/eureka	
zuul:
  routes:
    service-provider: /service-provider/** # 这里是映射路径

2.引导类加入@EnableZuulProxy，@EnableDiscoveryClient

3.也可加入路由前缀，发起请求时要用路有前缀开头

zuul:
  routes:
    service-provider: /service-provider/**
    service-consumer: /service-consumer/**
  prefix: /api # 添加路由前缀

2.5.2 过滤器

通过过滤器实现请求鉴权。

public abstract ZuulFilter implements IZuulFilter{

    abstract public String filterType();

    abstract public int filterOrder();
    
    boolean shouldFilter();// 来自IZuulFilter

    Object run() throws ZuulException;// IZuulFilter
}
  • shouldFilter``：返回一个Boolean`值，判断该过滤器是否需要执行。返回true执行，返回false不执行。
  • run：过滤器的具体业务逻辑。
  • filterType：返回字符串，代表过滤器的类型。包含以下4种：
    • pre：请求在被路由之前执行
    • route：在路由请求时调用
    • post：在route和errror过滤器之后调用
    • error：处理请求时发生错误调用
  • filterOrder：通过返回的int值来定义过滤器的执行顺序，数字越小优先级越高。

流程：

  • 请求到达首先会经过pre类型过滤器，而后到达route类型，进行路由，请求就到达真正的服务提供者，执行请求，返回结果后，会到达post过滤器。而后返回响应。

异常流程：

  • 整个过程中，pre或者route过滤器出现异常，都会直接进入error过滤器，在error处理完毕后，会将请求交给POST过滤器，最后返回给用户。
  • 如果是error过滤器自己出现异常，最终也会进入POST过滤器，将最终结果返回给请求客户端。
  • 如果是POST过滤器出现异常，会跳转到error过滤器，但是与pre和route不同的是，请求不会再到达POST过滤器了。

使用场景：

  • 请求鉴权：一般放在pre类型，如果发现没有访问权限，直接就拦截了
  • 异常处理：一般会在error类型和post类型过滤器中结合来处理。
  • 服务调用时长统计：pre和post结合使用。
